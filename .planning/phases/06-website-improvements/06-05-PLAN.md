<task type="auto">
  <name>Task 1: Dark mode polish — fix global styles, verify dividers and selection colors</name>
  <files>app/globals.css</files>
  <action>
Audit `app/globals.css` for dark mode compatibility and fix any issues found.

**Check and fix the following items:**

1. **Section dividers** — `.sectionDivider` uses `var(--color-accent)` and `.sectionDividerWide` uses `var(--color-border)` — both are CSS vars that respond to dark mode. Verify these are correct; if they render correctly in dark mode, no change needed.

2. **Selection color** — If no `::selection` rule exists, add one after the `body` block:
```css
::selection {
  background-color: var(--color-primary);
  color: var(--color-background);
}
```
This ensures text selection looks good in both modes.

3. **Scrollbar coloring** (optional polish) — Add after the selection rule if not present:
```css
/* Custom scrollbar for dark mode compatibility */
@supports (scrollbar-color: auto) {
  html {
    scrollbar-color: var(--color-text-muted) var(--color-background);
  }
}
```

4. **Verify no hardcoded colors** — Scan globals.css for any remaining hardcoded hex values or rgba() that should be CSS vars. The known exceptions that are OK to keep hardcoded:
   - Overlay gradients in service cards (self-contained, not theme-dependent)
   - White (#FFFFFF) for text on colored button backgrounds (Button.module.css)
   - The `@theme inline` block (resolved at build time — correct as-is)

Do NOT modify the `@theme inline` block, the `:root` block, or the `[data-theme]` blocks added in Plan 01.
